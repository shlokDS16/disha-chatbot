# Disha Chatbot — API Contract v1.0

**Status:** Draft for approval
**Scope:** Prototype — no auth, no Supabase, no n8n, no WhatsApp
**Last updated:** 2026-04-22

---

## 1. Overview

Disha is a multilingual genetic counselling chatbot. This document is the single source of truth for every HTTP endpoint the frontend calls, the exact JSON shapes, and the behaviour contract for each. Backend implements from this. Frontend builds against this. No room for mismatch.

### 1.1 Stack
- **Framework:** FastAPI (Python 3.11+)
- **Base URL (dev):** `http://localhost:8000`
- **Base path:** `/api/v1`
- **Content-Type:** `application/json` unless noted (multipart for file upload)
- **Auth (prototype):** Session-ID only (anonymous). Admin endpoints guarded by `X-Admin-Key`.
- **OpenAPI docs:** `/docs` (auto-generated by FastAPI)

### 1.2 Supported Languages
| Code | Native | English |
|------|--------|---------|
| `en` | English | English |
| `hi` | हिंदी | Hindi |
| `mr` | मराठी | Marathi |

### 1.3 Supported Diseases (extensible)
| ID | Name | Status |
|----|------|--------|
| `sickle_cell` | Sickle Cell Disease | active |
| `thalassemia` | Thalassemia | planned (enable via `/rag/ingest` + config flag) |

Adding a new disease = ingest chunks with its `disease` metadata + flip a config flag. **Zero code change.**

## 2. Global Conventions

### 2.1 Request Headers

| Header | When | Purpose |
|--------|------|---------|
| `X-Session-ID` | after `/session/start` | Current session UUID |
| `X-User-Language` | always | `en` / `hi` / `mr` |
| `X-Admin-Key` | admin endpoints | Shared secret from `.env` |
| `Content-Type` | always | `application/json` or `multipart/form-data` |

### 2.2 Standard Error Schema

```json
{
  "error": {
    "code": "INVALID_LANGUAGE",
    "message": "Unsupported language. Must be one of: en, hi, mr.",
    "details": { "received": "fr" }
  }
}
```

### 2.3 Error Codes

| HTTP | Code | Meaning |
|------|------|---------|
| 400 | `INVALID_INPUT` | Malformed body |
| 400 | `INVALID_LANGUAGE` | Language unsupported |
| 400 | `FILE_TOO_LARGE` | > 10 MB |
| 400 | `UNSUPPORTED_FILE_TYPE` | Not PDF / DOCX / image |
| 401 | `ADMIN_KEY_INVALID` | Missing/wrong admin key |
| 404 | `SESSION_NOT_FOUND` | Unknown session |
| 404 | `FILE_NOT_FOUND` | Unknown file ID |
| 409 | `CONSENT_REQUIRED` | Action needs user consent |
| 429 | `RATE_LIMITED` | Too many requests |
| 500 | `INTERNAL_ERROR` | Generic server error |
| 502 | `LLM_UNAVAILABLE` | Groq upstream failure |
| 502 | `OCR_UNAVAILABLE` | OCR.space upstream failure |
| 503 | `EMBEDDING_UNAVAILABLE` | HuggingFace Inference failure |

### 2.4 Rate Limits (prototype)
- `/chat/message`: 30 req/min per session
- `/files/upload`: 10 req/hour per session
- `/facilities/nearby`: 30 req/min per session

## 4. Content Block Catalog

Every chat response's `content` is an array of blocks. Frontend renders by `type`.

| `type` | `data` shape | UI |
|--------|-------------|----|
| `text` | `{ text: string }` | Markdown-capable bubble |
| `punnett` | `{ p1, p2, probabilities, grid }` | 3×3 genotype card |
| `facility` | `{ facilities[], user_location }` | Location card with map preview |
| `file_summary` | `{ file_id, key_findings[], what_it_means, next_steps[] }` | Three-section card (Seva Health style) |
| `crisis_banner` | `{ severity, helplines[], message }` | Amber banner above bubble |
| `consent_gate` | `{ action, scopes[], description }` | Inline mini-modal |
| `referral` | `{ facility_type, nearest[], why }` | "Go to Maps" CTA card |
| `starter_chips` | `{ chips: [{id, text}] }` | Empty-chat starter grid |

**Adding a block type:** add backend enum + frontend renderer. Nothing else.

## 5. Internal Chat Flow (inside `/chat/message`)

```
1. Validate session + fetch context
2. Detect language (if unset in session)
3. Distress classifier (lightweight local model)
   └─ high distress → inject crisis_banner block + iCall/Vandrevala
4. Embed query via BGE-M3 (HuggingFace Inference)
5. Hybrid search Qdrant (dense + sparse) → top-10
6. Rerank via BGE-reranker-v2-m3 → top-3
7. Route:
   ├─ top-1 rerank ≥ 0.78   → RAG only, no LLM
   ├─ 0.62 ≤ top-1 < 0.78   → LLM with top-3 as context
   └─ top-1 < 0.62          → LLM with system prompt only
8. If LLM:
   a. Assemble dynamic system prompt (§6)
   b. Call Groq (Llama 3.3 70B Versatile) with tool defs
   c. Handle tool_use loop (calculate_risk, find_facility, ...)
   d. Generate final text
9. If LLM path passed quality gate → auto-enrich Qdrant (§7)
10. Return structured response
```

**Thresholds tuned conservatively** — a false-positive RAG hit destroys trust faster than token cost ever will.

## 6. Dynamic System Prompt Assembly

Prompt is **composed per request**, not static. Blocks:

```
[BASE_IDENTITY]
Always. Disha, non-directive counsellor, knowledge boundaries.

[LANGUAGE_INSTRUCTION]
"Respond only in {mr|hi|en}. Use village-level vocabulary."

[AUDIENCE_TONE]
Switched by persona_hint + distress signal:
- asha_worker  → "Give a script the ASHA can read aloud."
- couple       → "Warm, reassuring, stories over statistics."
- patient      → "Address fear/guilt/acceptance explicitly."
- doctor       → "Clinical precision. Skip basic explanation."
- distressed   → "Empathy first. Short sentences. Offer support before info."

[RURAL_COMMUNICATION_STYLE]
Always. Plain words. Analogies (seeds, family tree). Avoid naked stats.
Confirm understanding ("Shall I explain with a story?").

[TOOL_AVAILABILITY]
Declares calculate_risk, find_facility, save_session (signatures).

[RAG_CONTEXT]
Included only if retrieval produced usable chunks:
"Verified reference material:
 [chunk 1] ...
 [chunk 2] ..."

[POLICY_CONTEXT]
From /policy/current: active guidelines to respect.

[SAFETY_RAILS]
Always. Never diagnose. Never prescribe. Never tell couples what to decide.
On distress → iCall 9152987821.
```

Implementation: a `SystemPromptBuilder` class composes blocks from runtime signals.

## 7. Auto-Enrichment

After LLM-path answer completes:

```python
if (
    len(answer_text) > 50
    and not contains_refusal(answer_text)
    and not contains_hallucinated_numbers(answer_text)
    and session.consent.allows("auto_rag_enrich")
):
    upsert_to_qdrant(
        text=f"Q: {query}\nA: {answer_text}",
        metadata={
            "disease": inferred_disease,
            "source": "auto_llm",
            "confidence": 0.70,
            "reviewed": False,
            "session_id": session.id,
            "language": session.language,
            "topic": classify_topic(query),
            "created_at": now,
        }
    )
```

Quality filter prevents garbage. `/rag/review` lets admin promote good auto-chunks to `confidence: 0.9+` or reject.

## 8. Multi-Disease Extensibility

**Single Qdrant collection** (`disha_knowledge`), chunks tagged with `disease` metadata. Qdrant payload index on `disease` keeps filtered search fast.

**To add thalassemia:**
1. `POST /rag/ingest` with `disease: "thalassemia"` + chunks — no code change
2. Update `/meta/diseases` config row → `active: true`
3. Add translations for disease name in UI i18n files

Total work: ~1 hour of content curation + translation.

## 9. Example Flows

### 9.1 New Marathi user, AS × AS question
1. `POST /session/start` → `{session_id}`
2. `POST /session/{id}/consent` → consent recorded
3. `POST /chat/message` body `{"text": "AS AS झाले तर?"}`
4. Backend: `lang=mr` → RAG hits `nhscp-p14` @ 0.89 → return RAG + Punnett card
5. Frontend renders text bubble + Punnett card + `📚` source pill
6. User taps `🔊` → Web Speech TTS reads aloud

### 9.2 Document upload
1. `POST /files/upload` (report.pdf) → `file_id`, `status=queued`
2. Frontend polls `GET /files/{id}` every 2s
3. Backend: OCR.space → `ocr_text` → LLM summarize in session language → `summary.key_findings`
4. Poll returns `status=done` with structured summary
5. Frontend renders `file_summary` card in chat
6. User asks "What should I do?" → `POST /chat/message` with `attachments: [{file_id}]` → LLM has OCR context

### 9.3 Find nearest HPLC centre
1. Frontend requests browser geolocation → `lat`/`lng`
2. `GET /facilities/nearby?lat=..&lng=..&service=HPLC_centre`
3. Backend calls Google Places Nearby → list with `directions_url`
4. Frontend renders facility cards → tap → opens Maps via `directions_url`
